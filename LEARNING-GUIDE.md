## 1. How the Full System Works (Big Picture)

When a user opens a browser and visits `http://localhost`, this is what happens:

```
Browser
  │
  ▼
Apache (port 80) ← WAMP manages this
  │
  ├── /api/*  ──────────► FastAPI (port 8000) ← Docker container
  │                           │
  │                           ├── PostgreSQL (port 5432) ← Docker container
  │                           └── Redis (port 6379)      ← Docker container
  │
  └── everything else ──► Next.js (port 3000) ← pm2 process
```

### Why this structure?

- **Apache** sits in front so users only see one address (`http://localhost`). No port numbers.
- **FastAPI** handles all data — it's the brain. It reads/writes to PostgreSQL and uses Redis for caching.
- **Next.js** handles the visual interface — it's what the user sees. It calls FastAPI to get data.
- **Docker** wraps the backend so it runs the same way on every machine.
- **pm2** keeps Next.js running as a background service.

## 2. Backend Source Code — FastAPI

### 2.1 Folder Structure and Why

```
app/
├── api/v1/          ← Route handlers (thin — just receive request, call service, return response)
├── core/            ← Config, database connection, security, shared dependencies
├── models/          ← Database table definitions (SQLAlchemy ORM)
├── schemas/         ← Request/response shapes (Pydantic)
├── services/        ← Business logic (rules, decisions, orchestration)
├── repositories/    ← Database queries (only place where SQL-like code lives)
├── middleware/      ← Code that runs on every request (logging, request IDs)
└── exceptions/      ← Custom error types and error response formatting
```

This is called **layered architecture**. Each layer has ONE job. This makes the code
easier to understand, test, and change.

### 2.2 The Flow of a Request (Example: Creating an Issue)

When a user creates an issue, the request passes through every layer:

```
POST /api/v1/projects/{project_id}/issues
         │
         ▼
   [Route Handler]  app/api/v1/issues.py
   - Validates the JWT token (via dependency)
   - Checks project membership (via dependency)
   - Hands off to service
         │
         ▼
   [Service]  app/services/issue_service.py
   - Checks: does the project exist?
   - Checks: is the epic valid (if provided)?
   - Checks: is the sprint active (if assigned to sprint)?
   - Calls repository to save
         │
         ▼
   [Repository]  app/repositories/issue_repository.py
   - Writes the SQLAlchemy INSERT query
   - Calls db.flush() to write to DB within the transaction
   - Returns the new Issue object
         │
         ▼
   [Schema]  app/schemas/issue.py → IssueResponse
   - Converts the SQLAlchemy model to JSON
   - Filters out internal fields
         │
         ▼
   HTTP 201 JSON response sent to browser
```

### 2.3 Models — The Database Tables

**File:** `app/models/user.py`

The model defines what the `users` table looks like:

```python
class User(Base, TimestampMixin):
    __tablename__ = "users"

    id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True), primary_key=True, server_default=text("gen_random_uuid()")
    )
    name: Mapped[str] = mapped_column(String(255), nullable=False)
    email: Mapped[str] = mapped_column(String(255), nullable=False, unique=True, index=True)
    password_hash: Mapped[str] = mapped_column(Text, nullable=False)
    system_role: Mapped[SystemRole] = mapped_column(
        Enum(SystemRole, name="system_role_enum"), nullable=False, default=SystemRole.developer
    )
    is_active: Mapped[bool] = mapped_column(Boolean, nullable=False, default=True)
```

**Why UUID instead of integer IDs?**
- UUIDs are safe to expose in URLs — they don't reveal how many records exist
- They can be generated by the database without coordination (`gen_random_uuid()`)
- Example in your project: `GET /api/v1/projects/3f7e1c2a-0000-0000-0000-000000000001`

**Why `is_active` instead of deleting users?**
- If you delete a user, you lose history (who created that issue? who wrote that comment?)
- Soft delete = mark as inactive, keep the data
- When listing users, the service filters `WHERE is_active = true`

**Why `password_hash` and never `password`?**
- Passwords are hashed with bcrypt before storing
- Even if someone steals the database, they can't recover the passwords
- bcrypt is a one-way function — you can verify "is Admin123 correct?" but you cannot reverse it to "Admin123"

### 2.4 Schemas — What Goes In and Out

**File:** `app/schemas/user.py`

Schemas define what the API accepts and what it returns. Every entity has three schemas:

```python
# What the frontend sends when creating a user
class UserCreate(BaseModel):
    name: str
    email: str
    password: str  # raw password — service hashes it before saving

# What the frontend sends when updating a user
class UserUpdate(BaseModel):
    name: str | None = None      # optional — only update what's provided
    email: str | None = None

# What the API returns — notice: no password_hash!
class UserResponse(BaseModel):
    model_config = ConfigDict(from_attributes=True)  # converts ORM model to dict
    id: UUID
    name: str
    email: str
    system_role: SystemRole
    is_active: bool
    created_at: datetime
```

**Why three schemas per entity?**
- `Create` has required fields and the raw password
- `Update` has all optional fields (PATCH — only change what you send)
- `Response` hides internal fields (`password_hash`) and adds computed fields

**Why `model_config = ConfigDict(from_attributes=True)`?**
- Without this, Pydantic can't read SQLAlchemy objects (they're not regular dicts)
- With it, Pydantic reads the ORM object's attributes like a dict
- Example: `UserResponse.model_validate(user_orm_object)` → works

### 2.5 Services — The Business Logic

**File:** `app/services/issue_service.py`

Services contain all the rules. Example — changing an issue's status:

```python
async def change_status(self, issue_id: UUID, new_status: IssueStatus, current_user: User) -> Issue:
    issue = await self.issue_repo.get_by_id(issue_id)
    if not issue:
        raise NotFoundError("Issue not found", {"issue_id": str(issue_id)})

    # Business rule: not every status change is allowed
    ALLOWED_TRANSITIONS = {
        IssueStatus.todo: [IssueStatus.in_progress],
        IssueStatus.in_progress: [IssueStatus.in_review, IssueStatus.todo],
        IssueStatus.in_review: [IssueStatus.done, IssueStatus.in_progress],
        IssueStatus.done: [IssueStatus.in_review],
    }
    if new_status not in ALLOWED_TRANSITIONS[issue.status]:
        raise ConflictError(f"Cannot move from {issue.status} to {new_status}")

    # Business rule: developers can only update their own issues
    if current_user.system_role != SystemRole.admin:
        if project_member.project_role == ProjectRole.developer:
            if issue.assignee_id != current_user.id:
                raise ForbiddenError("Developers can only update their own issues")

    # All checks passed — update
    return await self.issue_repo.update(issue_id, {"status": new_status})
```

**Why services don't call other services:**
- If `IssueService` called `SprintService`, and `SprintService` called `IssueService`,
  you'd get a circular import that crashes Python
- Instead, if `IssueService` needs sprint data, it uses `SprintRepository` directly

### 2.6 Repositories — The Database Queries

**File:** `app/repositories/issue_repository.py`

Repositories are the ONLY place where database queries are written:

```python
class IssueRepository(BaseRepository[Issue]):
    model = Issue

    async def list_by_project(
        self,
        project_id: UUID,
        status: IssueStatus | None = None,
        page: int = 1,
        limit: int = 25,
    ) -> tuple[list[Issue], int]:
        query = select(Issue).where(Issue.project_id == project_id)
        if status:
            query = query.where(Issue.status == status)
        # Count total for pagination
        count_result = await self.db.execute(select(func.count()).select_from(query.subquery()))
        total = count_result.scalar_one()
        # Get the page
        result = await self.db.execute(query.offset((page - 1) * limit).limit(limit))
        return result.scalars().all(), total
```

**Why only repositories write queries?**
- If queries were scattered in services and routes, finding and fixing a slow query would mean
  searching the whole codebase
- With repositories, all queries for `Issue` are in `issue_repository.py` — one place

**Why `flush()` and not `commit()`?**
- `flush()` sends the SQL to the database within the current transaction but doesn't finalize it
- `commit()` finalizes everything — if one operation fails after a `commit()`, you can't roll back
- The `get_db()` dependency in `app/core/dependencies.py` wraps each request in one transaction
  and calls `commit()` at the end — or `rollback()` if anything raised an exception

### 2.7 Authentication — How JWT Works

**File:** `app/core/security.py`

When a user logs in:

```
1. User sends: { email: "admin@admin.com", password: "Admin123" }
2. Server checks: bcrypt.verify("Admin123", stored_hash) → True
3. Server creates access token (15 min): JWT { sub: user_id, type: "access" }
4. Server creates refresh token (7 days): random string, SHA-256 hash stored in DB
5. Server returns: { access_token: "eyJ...", refresh_token: "abc123..." }
```

For every subsequent request:
```
1. Browser sends: Authorization: Bearer eyJ...
2. Server decodes JWT → gets user_id
3. Server loads user from DB → checks is_active = true
4. If valid → request proceeds
5. If expired → 401, frontend must use refresh token to get new access token
```

**Why store only the hash of the refresh token?**
- The raw refresh token is like a password — if the DB is stolen, attackers can't use hashes
- The server re-hashes the token on each `/auth/refresh` request and compares

## 4. Backend CI/CD Pipeline

### 4.1 What CI/CD Means in Simple Terms

**CI (Continuous Integration):** Every time you push code, a machine automatically checks it.
If the check fails, you know before it reaches production.

**CD (Continuous Deployment):** When code is merged to `main`, a machine automatically
deploys it to the server. No manual steps.

Without CI/CD: developer pushes broken code → it reaches production → users see errors.
With CI/CD: developer pushes broken code → CI catches it → developer fixes it → safe to deploy.

### 4.2 Backend CI — What Runs and Why

**File:** `.github/workflows/ci.yml`

```yaml
name: CI
on:
  push:
    branches: ["**"]     # runs on EVERY branch push
  pull_request:
    branches: ["**"]

jobs:
  lint-and-test:
    runs-on: ubuntu-latest   # runs on GitHub's cloud machines (free)
```

**Step 1 — black (code formatter)**

```yaml
- name: Check formatting (black)
  run: black --check --diff app/ tests/ main.py
```

What it does: black rewrites your Python code to follow a consistent style.
`--check` means "don't rewrite, just fail if it would change anything."

Example from your project — before black:
```python
def create_epic(project_id,data,current_user):
    return await self.epic_repo.create({"project_id":project_id,"title":data.title})
```

After black:
```python
def create_epic(project_id, data, current_user):
    return await self.epic_repo.create({"project_id": project_id, "title": data.title})
```

Why: Every developer's code looks the same. Code reviews focus on logic, not style debates.

---

**Step 2 — isort (import sorter)**

```yaml
- name: Check import order (isort)
  run: isort --check-only --diff app/ tests/ main.py
```

What it does: sorts Python imports in a consistent order: stdlib → third-party → local.

Example from your `main.py`:
```python
# After isort — correct order:
import logging                          # stdlib

from fastapi import FastAPI             # third-party
from fastapi.exceptions import ...

from app.api.v1.router import v1_router # local (your code)
from app.core.config import settings
```

Why: Without isort, different developers add imports wherever. Hard to read, merge conflicts.

---

**Step 3 — ruff (linter)**

```yaml
- name: Lint (ruff)
  run: ruff check app/ tests/ main.py
```

What it does: scans the code for bugs and bad patterns — without running it.

Real examples of things ruff catches in your project:
```python
# F401 — imported but never used (wastes memory, confuses readers)
from app.models.epic import Epic    # if Epic is never used in the file

# B006 — mutable default argument (common Python bug)
def create_issues(items=[]):   # WRONG — this list is shared between all calls
def create_issues(items=None): # CORRECT

# UP007 — old-style type hints
def get_user(id: Optional[UUID])  # old
def get_user(id: UUID | None)     # modern Python 3.12 syntax
```

Why ruff specifically (and not pylint or flake8)?
- ruff is 10–100x faster than older linters
- It replaces flake8, pylint, pyupgrade, and isort rules in one tool
- Written in Rust — finishes in milliseconds on your codebase

### 4.3 Backend CD — What Runs and Why

**File:** `.github/workflows/cd.yml`

```yaml
on:
  push:
    branches: ["main"]    # ONLY runs when merging to main
  workflow_dispatch:       # also allows manual trigger from GitHub UI

jobs:
  deploy:
    runs-on: self-hosted   # runs on YOUR machine, not GitHub's cloud
    environment: production
```

**Why `self-hosted`?**
- GitHub's cloud machines can't reach `localhost:8000` on your Windows machine
- The self-hosted runner IS your Windows machine — it runs inside `C:\actions-runner`
- When a push to main happens, GitHub sends a job to your runner, which executes it locally

---

**CD Step 1 — Write the .env file**

```yaml
- name: Write .env file
  run: |
    cat > .env << 'ENVEOF'
    DATABASE_URL=postgresql+asyncpg://jira:${{ secrets.DB_PASSWORD }}@db:5432/jira_db
    SECRET_KEY=${{ secrets.SECRET_KEY }}
    CORS_ORIGINS=["http://localhost","http://localhost:80"]
    ENVEOF
```

The secrets (`DB_PASSWORD`, `SECRET_KEY`) are stored in GitHub → Settings → Secrets.
They are never in the code. The CD workflow writes them to `.env` at deploy time.

Why not commit `.env`?
- `.env` contains passwords and secret keys
- If committed to GitHub, anyone with repo access (or who forks it) gets your secrets
- GitHub Secrets are encrypted and only injected at runtime

---

**CD Step 2 — Build Docker image**

```yaml
- name: Build Docker image
  run: docker compose -f docker-compose.yml -f docker-compose.prod.yml build --no-cache app
```

`docker compose build` reads your `Dockerfile` and creates a new image.

Your `Dockerfile` has two stages:

```dockerfile
# Stage 1 — builder: install Python packages
FROM python:3.12-slim AS builder
RUN pip install --no-cache-dir --prefix=/install .

# Stage 2 — runtime: only copy what's needed to run
FROM python:3.12-slim AS runtime
COPY --from=builder /install /usr/local
COPY . .
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

Why two stages?
- Stage 1 needs gcc and build tools (large)
- Stage 2 only needs the compiled packages (small)
- Final image is ~200MB instead of ~800MB

Why `--no-cache`?
- Without it, Docker reuses old layers — if a dependency changed but the layer is cached,
  you'd run old code in production
- With `--no-cache`, every build is fresh

---

**CD Step 3 — Run deploy.sh**

```yaml
- name: Run deployment script
  run: bash scripts/deploy.sh
```

`scripts/deploy.sh` does five things:

```bash
# 1. Start db and redis (if not already running)
docker compose up -d db redis

# 2. Wait for PostgreSQL to be ready (polls pg_isready up to 20 times)
for i in $(seq 1 20); do
    if docker compose exec -T db pg_isready -U jira -d jira_db; then
        break
    fi
    sleep 3
done

# 3. Run Alembic migrations
docker compose run --rm app alembic upgrade head

# 4. Start the app container (--no-deps = don't restart db/redis)
docker compose up -d --no-deps app

# 5. Clean up old Docker images
docker image prune -f
```

Why Alembic migrations in CD?
- When you add a new column to a model, the database table doesn't update automatically
- Alembic tracks schema changes in `alembic/versions/` and applies them in order
- Example: you add `due_date` to the `issues` table → Alembic generates:
  ```python
  def upgrade():
      op.add_column("issues", sa.Column("due_date", sa.Date(), nullable=True))
  def downgrade():
      op.drop_column("issues", "due_date")
  ```
- `alembic upgrade head` applies all pending migrations automatically on deploy

Why `--no-deps` when starting the app?
- `docker compose up -d app` would restart db and redis too (because app depends on them)
- We already started db and redis in step 1 — restarting them would lose in-flight connections
- `--no-deps` starts only the `app` container

---

**CD Step 4 — Health check**

```yaml
- name: Health check
  run: |
    for i in $(seq 1 12); do
      STATUS=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:8000/health)
      if [ "$STATUS" = "200" ]; then
        echo "App is healthy"
        exit 0
      fi
      sleep 5
    done
    exit 1  # fail the CD workflow if app never became healthy
```

Polls `GET /health` every 5 seconds for up to 60 seconds. If it never returns 200:
- The CD workflow fails with a red ✗ on GitHub
- The deploy script prints the last 50 lines of Docker logs
- You can see exactly what went wrong

Why a health check?
- Without it, the CD workflow would succeed even if the app crashed on startup
- With it, you know within 60 seconds if the deployment actually worked

### 5.1 Frontend CI — What Runs and Why

**Step 1 — pnpm install with frozen lockfile**

```yaml
- name: Install dependencies
  run: pnpm install --frozen-lockfile
```

Why `--frozen-lockfile`?
- `pnpm-lock.yaml` records the exact version of every package (including sub-dependencies)
- Without it, `pnpm install` might upgrade a package that breaks the build
- `--frozen-lockfile` says "install EXACTLY what's in the lock file, fail if it's outdated"

Why pnpm instead of npm or yarn?
- pnpm uses a global content-addressable store — if `react@19.2.4` is installed in two projects,
  it's stored once on disk and hard-linked in both `node_modules/`
- Your project has ~800MB of `node_modules/` — with pnpm this is much lighter

Prettier reformats TypeScript, TSX, CSS, and JSON to a consistent style.

Example from your `app/projects/page.tsx` — before Prettier:
```tsx
export default function ProjectsPage(){
```

Your `.prettierrc` config:

```json
{

  "singleQuote": false,   ← use double quotes (matches JSX)
  "tabWidth": 2,          ← 2 spaces indent
  "trailingComma": "es5", ← trailing commas where valid in ES5
```

Why `prettier-plugin-tailwindcss`?
- Tailwind has a recommended class order (layout → box model → typography → etc.)
- Without this plugin, developers write classes in random order
- With it, Prettier sorts them automatically — `p-4 flex gap-2` becomes `flex gap-2 p-4`

**CD Step 2 — Run deploy.sh**

`scripts/deploy.sh` does three things:

```bash
# 1. Install dependencies (exact versions from lock file)
pnpm install --frozen-lockfile

# 2. Build the production Next.js bundle
pnpm build
# This creates: .next/  folder with optimized JS/CSS bundles

# 3. Restart (or start) the pm2 process
if pm2 describe jira-frontend > /dev/null 2>&1; then
    pm2 restart jira-frontend   # already exists → just restart
else
    pm2 start pnpm --name jira-frontend -- start   # first time → create
fi

pm2 save   # persist so it survives Windows reboot
```

Why pm2?
- Next.js is a Node.js HTTP server — it needs to keep running as a background process
- Without pm2, you'd need a terminal open 24/7 running `pnpm start`
- pm2 keeps it running, restarts it if it crashes, and survives reboots

**CD Step 3 — Health check**

```yaml
- name: Health check
  run: |
    for i in $(seq 1 12); do
      STATUS=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:3000)
      if [ "$STATUS" = "200" ] || [ "$STATUS" = "301" ] || [ "$STATUS" = "307" ]; then
        exit 0
      fi
      sleep 5
    done
    exit 1
```

Accepts 200, 301, or 307 because Next.js redirects `/` to `/auth/login` for unauthenticated
requests. Any of these means the server is up.

## 6. Apache Reverse Proxy

**File:** `D:\wamp64\alias\jira-api.conf`

```apache
ProxyPass        /api/    http://127.0.0.1:8000/api/
ProxyPass        /docs    http://127.0.0.1:8000/docs
ProxyPass        /health  http://127.0.0.1:8000/health
ProxyPass        /        http://127.0.0.1:3000/    ← MUST be last (catch-all)
```

**Why order matters:** Apache checks these rules top to bottom. The catch-all `/` must be LAST.
If it were first, every request (even `/api/`) would go to Next.js.

**Why Apache at all?**
Without Apache, users would need to know ports:
- `http://localhost:3000` for the frontend
- `http://localhost:8000` for the backend

With Apache, everything is on `http://localhost`.

**How `ProxyPassReverse` works:**
When FastAPI returns a redirect `Location: http://127.0.0.1:8000/docs`, Apache rewrites it to
`Location: http://localhost/docs` so the browser doesn't get confused by internal addresses.

## 7. How a Request Travels Through the Whole System

### Example: User opens the Issues page for a project

```
1. User clicks "Issues" in the nav
   Browser requests: GET http://localhost/projects/abc-123/issues/xyz-456

2. Apache receives the request on port 80
   Rule: /projects/* doesn't match /api/, /docs, /health
   → Forwards to Next.js: GET http://127.0.0.1:3000/projects/abc-123/issues/xyz-456

3. Next.js renders the page component:
   app/projects/[id]/issues/[issueId]/page.tsx
   The component fetches data:
   GET http://localhost:8000/api/v1/projects/abc-123/issues/xyz-456
   (with Authorization: Bearer eyJ... header)

4. FastAPI receives the API request on port 8000
   Route: GET /api/v1/projects/{project_id}/issues/{issue_id}
   Handler: app/api/v1/issues.py → get_issue()

5. get_current_user dependency decodes the JWT → User object
   get_project_member dependency loads ProjectMember → checks access

6. IssueService.get_issue() is called
   → IssueRepository.get_with_comments(issue_id) runs the DB query
   → PostgreSQL (Docker container) returns the row
   → SQLAlchemy maps it to an Issue ORM object

7. FastAPI serializes Issue → IssueResponse (Pydantic schema)
   Returns: HTTP 200 { "id": "xyz-456", "title": "Fix login bug", ... }

8. Next.js component receives the JSON
   Renders the issue detail UI

9. Browser displays the page to the user
```

Total time for this round-trip: typically 50–200ms on localhost.
